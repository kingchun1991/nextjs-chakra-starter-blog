# Internationalization (i18n) Guide

This blog template comes with optional internationalization (i18n) support using [next-intl](https://next-intl-docs.vercel.app/). **By default, i18n is disabled** for a simpler single-language experience. You can easily enable it when you need multilingual support.

## Quick Overview

**Key Features:**
- 🎯 **Disabled by default** - Simple single-language setup out of the box
- 🌍 Optional multi-language support when you need it
- 🔄 Easy language switching with a header button (when enabled)
- 📝 Locale-based content organization
- 🎯 SEO-friendly locale routing
- ⚡ Zero configuration required for English-only blogs

## Enabling Internationalization

To enable multi-language support:

### 1. Create Environment File

Create a `.env.local` file in the project root (you can optionally copy from the example):

```bash
cp .env.local.example .env.local
```

**Note:** The `.env.local` file is **completely optional**. Without it, the template defaults to English-only mode.

### 2. Enable i18n

Add this line to `.env.local`:

```bash
NEXT_PUBLIC_ENABLE_I18N=true
```

### 3. Restart the Development Server

After enabling or disabling i18n, restart your development server:

```bash
pnpm dev
```

That's it! Your blog now supports English and Traditional Chinese. The language switcher will appear in the header automatically.

## How It Works

The template uses **Next.js rewrites** to provide clean URLs when i18n is disabled:

### When i18n is ENABLED (`NEXT_PUBLIC_ENABLE_I18N=true`)

1. **Locale-based Routing**: All routes use the `[locale]` directory structure (e.g., `/en/blog` or `/zh-TW/blog`)
2. **Content Organization**: Blog posts are organized by locale in `content/blog/[locale]/`
3. **UI Translations**: Interface text is loaded from `messages/[locale].json`
4. **Language Switcher**: Appears in the header for switching between languages
5. **Middleware**: Handles locale detection and routing with next-intl

### When i18n is DISABLED (default)

1. **Clean URLs**: Routes work without locale prefixes (e.g., `/blog` instead of `/en/blog`)
2. **URL Rewrites**: Next.js internally rewrites `/blog` to `/en/blog` without exposing it to users
3. **Single Language**: Only English content from `content/blog/en/` is used
4. **No Language Switcher**: Language switcher is hidden
5. **Simpler Configuration**: No locale routing overhead

**Note:** Simply restart your development server after toggling the i18n flag - no rebuild required!

## Supported Languages

By default, when i18n is enabled, this template supports:

- **English (en)** - Default language
- **Traditional Chinese (zh-TW)**

You can easily add more languages by following the steps below.

## Adding New Languages

To add a new language (e.g., Spanish):

### 1. Update Routing Configuration

Edit `src/i18n/routing.ts`:

```typescript
export const routing = defineRouting({
  locales: ['en', 'zh-TW', 'es'], // Add 'es' for Spanish
  defaultLocale: 'en',
  localePrefix: 'as-needed',
});
```

### 2. Create Translation Files

Create a new message file `messages/es.json`:

```json
{
  "header": {
    "home": "Inicio",
    "blog": "Blog",
    "tags": "Etiquetas",
    "about": "Acerca de"
  },
  "blog": {
    "readMore": "Leer más",
    "backToBlog": "Volver al blog"
  }
}
```

### 3. Create Content Directory

Create a new directory for Spanish blog content:

```bash
mkdir -p content/blog/es
```

### 4. Add Translated Content

Copy your blog posts to the new directory and translate them:

```bash
cp content/blog/en/get-started.mdx content/blog/es/get-started.mdx
# Then translate the content in the Spanish version
```

### 5. Update Language Switcher (Optional)

If you want to customize the language switcher labels, edit `src/lib/layout/language-switcher.tsx`.

## Disabling Internationalization

To disable i18n and use only English:

### Option 1: Using Environment Variable (Recommended)

i18n is **disabled by default** when:
- No `.env.local` file exists, OR
- `NEXT_PUBLIC_ENABLE_I18N` is not defined, OR
- `NEXT_PUBLIC_ENABLE_I18N=false` is set

Simply remove the `.env.local` file or remove/change the variable to disable i18n.

### Option 2: Remove i18n Completely

If you never plan to use i18n:

1. Uninstall next-intl:
```bash
pnpm remove next-intl
```

2. Remove i18n directories:
```bash
rm -rf src/i18n messages content/blog/zh-TW
```

3. Update imports in your code to remove next-intl references

## Translating Content

### Blog Posts

Each blog post should have a version in each supported language:

```
content/
└── blog/
    ├── en/
    │   ├── my-post.mdx
    │   └── another-post.mdx
    └── zh-TW/
        ├── my-post.mdx
        └── another-post.mdx
```

**Important:** The filename should be the same across languages (e.g., `my-post.mdx`), but the content inside can be fully translated.

### UI Elements

Translate interface text in the message files:

**messages/en.json:**
```json
{
  "header": {
    "home": "Home",
    "blog": "Blog"
  }
}
```

**messages/zh-TW.json:**
```json
{
  "header": {
    "home": "首頁",
    "blog": "部落格"
  }
}
```

Then use them in components:

```tsx
import { useTranslations } from 'next-intl';

export function MyComponent() {
  const t = useTranslations('header');

  return <h1>{t('home')}</h1>;
}
```

## Language Switcher

When i18n is enabled, the language switcher appears in the header and allows users to toggle between available languages. It:

- 🌐 Shows a globe icon
- 🔄 Preserves the current page when switching languages
- 📍 Updates the URL with the new locale prefix
- ✨ Has a smooth transition with Chakra UI styling
- 🎯 Automatically hides when i18n is disabled

You can customize its appearance in `src/lib/layout/language-switcher.tsx`.

## Best Practices

1. **Start Simple**: Begin with i18n disabled (single language) and enable when needed
2. **Consistent Structure**: Keep the same file structure across all locales
3. **SEO**: Each language gets its own URL, which is great for SEO
4. **Fallback**: The template falls back to English if content in a locale is missing
5. **Testing**: Always test language switching to ensure content is properly translated
6. **Easy Upgrades**: Template users can upgrade to newer versions without breaking i18n changes

## Technical Details

- **Library**: next-intl v4.8.3
- **Routing**: App Router with dynamic `[locale]` segment
- **Middleware**: Conditional middleware based on environment variable
- **Content Loading**: Locale-aware MDX file loading
- **Default Behavior**: i18n disabled (English only) for simplicity
- **Configuration**: Single environment variable (`NEXT_PUBLIC_ENABLE_I18N`)
